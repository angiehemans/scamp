# Fonts — Plan

**Status:** Proposed
**Date:** 2026-04-16
**Backlog item:** Not yet in `docs/backlog.md` — add once scoped.

## Goal

Replace the hardcoded list of ~19 web-safe and Google-popular font
families in the Typography section with a real font library the user
actually owns:

- **Phase 0.5** — new Project Settings page that replaces the
  Toolbar's global Settings button. Lives in the project folder,
  holds per-project config (starting with artboard background),
  and is the home for the Fonts panel that lands in Phase 2.
- **Phase 1** — enumerate the user's locally installed system fonts
  and show those in the font picker via a plain `<select>`. Every
  applied font is written with a sensible fallback so generated CSS
  still works on machines without the font installed.
- **Phase 1.5** — replace the `<select>` with a typeahead combobox
  so font pickers with 2000+ entries stay usable.
- **Phase 2** — let the user paste a Google Fonts embed link into
  the Project Settings → Fonts section. The `@import` URL is
  appended to `theme.css`, the fonts load on the canvas preview,
  and the project's own theme file carries them everywhere the
  project goes.

## Resolved open questions (from review)

- **Per-project settings landing place.** Fonts belong next to
  other per-project configuration, not in the global settings
  menu. A new Project Settings page ships as Phase 0.5 and becomes
  the parent surface for Phase 2's Fonts panel and any future
  project-scoped config.
- **Autoload.** Every new project already gets a `theme.css` and
  loads it at open (see `src/main/ipc/project.ts` — `createProject`
  writes `theme.css`, `openProject` backfills it if missing).
  Fonts piggyback on that same file: Google Fonts `@import` URLs
  go at the top of `theme.css`, colour and (future) font-family
  tokens live in the `:root` block below. No new per-project
  config file for fonts.
- **Font tokens in theme.css.** Defining `--headings: "Playfair
  Display", serif` as a token is explicitly desirable but is a
  separate follow-up — it's a ThemePanel extension, not a Fonts
  panel extension. Flagged in "Future work" so we don't lose it.
- **Phase 1 picker fidelity.** Phase 1 ships with a native
  `<select>` to get the system-font library usable as fast as
  possible. Phase 1.5 replaces it with a typeahead.

---

## Current state

- `TypographySection.tsx` hardcodes a static `FONT_FAMILY_OPTIONS`
  array with 19 entries. Each value already includes a fallback
  (`'Inter, sans-serif'`).
- `ScampElement.fontFamily` is `string | undefined` — already
  free-form, so the underlying model doesn't change.
- `generateCode.ts` emits `font-family: ${el.fontFamily};` verbatim.
- `cssPropertyMap.ts` round-trips `font-family` as an opaque string.
- `ElementRenderer.tsx` applies `fontFamily` as an inline style on
  text elements during canvas preview.
- `theme.css` is created on `ProjectCreate` with `DEFAULT_THEME_CSS`
  and backfilled on `ProjectOpen` if missing. `ThemePanel` reads
  and writes the file; `parseTheme.ts` currently extracts `--var`
  tokens from `:root`.
- `Settings` (global) holds `defaultProjectsFolder` and
  `artboardBackground` in Electron's userData dir. The
  `artboardBackground` value is passed from `ProjectShell` into
  `Viewport` as a prop.
- The Toolbar renders a settings gear that opens the global
  `SettingsPage` via `App.tsx`. `StartScreen` opens the same page.

Gaps:

- No per-project settings surface — every setting today is either
  global (Electron userData) or hidden in files (theme.css).
- `artboardBackground` is global, but the user wants it per-project.
- No system-font enumeration — nothing queries the OS font list.
- No canvas-side loader for non-system fonts, so Phase 2 font
  imports wouldn't actually render in preview without new wiring.
- `parseTheme` ignores `@import` lines, so Phase 2 needs the
  parser to preserve them on round-trip even though we don't
  turn them into tokens.

# Phase 0.5 — Project Settings page

## Goal

Give each project its own settings surface, reachable from a
Toolbar button in `ProjectShell`. The global Settings page stays
accessible from `StartScreen` only — inside a project, the gear
points to the project page.

## Scope for 0.5

One setting migrated: `artboardBackground`. Everything else is
additive scaffolding for Phase 2 (Fonts section) and future
per-project config (snap grid, default font, etc.).

## Where per-project settings live

Persist in `scamp.config.json` at the project root, alongside
`theme.css` and `agent.md`:

```json
{
  "artboardBackground": "#0f0f0f"
}
```

Why JSON and not CSS:

- Not every setting is a CSS concept. Artboard background isn't
  emitted to any generated file — it's a canvas-preview-only
  value. Dropping it in `theme.css` would mean either exporting a
  CSS var the user never uses, or defining a private convention
  inside a user-visible file. A typed JSON blob is cleaner.
- Future settings (snap grid, default element names, keybinding
  overrides) are also CSS-irrelevant.

The file is human-readable, checked-in-to-git-safe, and easy for
agents to touch if needed. Missing file → treat as defaults.
Missing key → use the default from `DEFAULT_PROJECT_CONFIG`.

## Data flow

### New IPC

| Channel | Payload |
|---|---|
| `projectConfig:read` | `{ projectPath } → ProjectConfig` |
| `projectConfig:write` | `{ projectPath, config } → ProjectConfig` |

Handlers in `src/main/ipc/projectConfig.ts`. Read returns defaults
for any missing/malformed key (same forgiving pattern as the
global settings handler). Write is a full replace — callers pass
the full desired object.

`ProjectConfig` type in `src/shared/types.ts`:

```ts
export type ProjectConfig = {
  artboardBackground: string;
};

export const DEFAULT_PROJECT_CONFIG: ProjectConfig = {
  artboardBackground: '#0f0f0f',
};
```

### Load at project open

`ProjectShell` already fetches settings on mount. Add a parallel
fetch for the project config. Replace the `settings?.artboardBackground`
wiring with `projectConfig.artboardBackground`. Keep global
`settings` fetched for anything else the Toolbar might need later
(today nothing else lives there for project context).

### Project creation

`createProject` writes an initial `scamp.config.json` with the
default values. `openProject` backfills a missing file with the
defaults — same shape as the existing `theme.css` backfill.

### Migration

The global `Settings.artboardBackground` field is no longer used
inside a project. Keep it in the global `Settings` type for one
release so old installs don't crash on a missing key, then remove
in a follow-up. The global Settings page stops rendering the
artboard-background row as part of this phase.

## UI

### New `ProjectSettingsPage` component

`src/renderer/src/components/ProjectSettingsPage.tsx` — full-page
view overlaid on `ProjectShell` (similar to how `SettingsPage`
overlays `StartScreen` today via the `App`-level view switch).
Takes a `projectPath` and an `onBack` callback.

Sections:

- **General**
  - Artboard background (ColorInput)
- **Fonts** — placeholder subsection added in Phase 2.

Save behaviour: patch-on-change, like the global settings page
does today. Every control writes the full config back via
`projectConfig:write`.

### Toolbar button rewire

- Inside `ProjectShell`, the gear icon now opens
  `ProjectSettingsPage` instead of the global `SettingsPage`.
- Add a small "Global settings" link on the `ProjectSettingsPage`
  so the route isn't lost for users who expect it — or leave it
  StartScreen-only and add later if users ask.
- `StartScreen` continues to open the global `SettingsPage`
  unchanged.

### View wiring

`App.tsx` grows a `'projectSettings'` view that needs the current
project's path. Easiest approach: when `ProjectShell` opens
project settings, it swaps its own body for `<ProjectSettingsPage>`
(no App-level view switch, no prop drilling). Matches how
`ThemePanel` already overlays inside `ProjectShell`.

## Implementation phases (0.5)

### 1. Project config backend

1. `ProjectConfig` + `DEFAULT_PROJECT_CONFIG` in `src/shared/types.ts`.
2. `IPC.ProjectConfigRead` + `IPC.ProjectConfigWrite`.
3. `src/main/ipc/projectConfig.ts` with read/write handlers.
4. Extend `project.ts:createProject` to write
   `scamp.config.json`. Extend `openProject` to backfill.
5. Preload exposure (`readProjectConfig`, `writeProjectConfig`).

### 2. Wire `artboardBackground` through project config

1. `ProjectShell` fetches project config on mount.
2. Pass `projectConfig.artboardBackground` into `Viewport`.
3. Remove `artboardBackground` as the data source from global
   `SettingsPage` (leave type field for one release, no UI).

### 3. ProjectSettingsPage UI

1. New component + module CSS, matching the existing
   `SettingsPage` visual language.
2. Render General → Artboard background.
3. Wire the Toolbar gear inside `ProjectShell` to a local
   `showProjectSettings` toggle that renders the page over the
   shell body.

### 4. Tests

- `test/projectConfig.test.ts` — validator tolerates missing keys,
  wrong types, malformed JSON; default values are returned in each
  case.

**Acceptance (0.5):** creating a new project writes
`scamp.config.json` with the defaults; opening an old project
without one backfills it; changing artboard background from the
Toolbar gear persists per-project; global settings page no
longer shows the artboard control; two projects with different
backgrounds don't leak into each other.

# Phase 1 — Local system fonts

## Data flow

### Enumerating fonts

Chromium (and therefore Electron's renderer) exposes the Local Font
Access API as `window.queryLocalFonts()`. It returns
`FontData[]` with `family`, `fullName`, `postscriptName`, and
`style`. It requires the `local-fonts` permission, which we grant
from the main process for Scamp's own window.

Enumeration flow:

1. On app start, call `window.queryLocalFonts()`.
2. De-duplicate by `family` — one entry per family, not per weight.
3. Sort alphabetically.
4. Cache in a Zustand slice (`fontsSlice`) so every Typography
   section instance reuses the same list without re-querying.

The call is a Promise and is cheap on warm systems but can be
noticeable on first launch with 2000+ fonts. The picker remains
usable (hardcoded fallback list) until data lands.

### Falling back when the API isn't available

`queryLocalFonts` is gated behind a permission and an origin check.
Electron renderers loaded from `file://` sometimes fall outside the
allowed origins. Guard the call:

```ts
if (typeof window.queryLocalFonts !== 'function') {
  return []; // fall back to the hardcoded common list
}
```

If the API returns an empty array or throws, fall back to the
current hardcoded list so the user can still pick something. Log a
warning.

### Fallback stacks on apply

When the user picks `Helvetica Neue` from the picker, we store the
quoted family name plus an inferred category fallback so the
resulting CSS still renders on someone else's machine:

```
font-family: "Helvetica Neue", sans-serif;
```

Inference (pure, no external API):

- family name contains `Mono`, `Code`, `Courier`, `Consolas` →
  `monospace`
- family name contains `Serif`, `Times`, `Georgia`, `Garamond`,
  `Playfair`, `Merriweather` → `serif`
- family name contains `Script`, `Hand`, `Brush` → `cursive`
- everything else → `sans-serif`

This heuristic is best-effort and only runs at pick time. The
user's stored string is authoritative after that — hand-edits
round-trip unchanged.

### Storage format

`el.fontFamily` remains a free-form string. When the picker sets
it, we store the full `"Family Name", fallback` expression. Family
names containing spaces or punctuation are quoted:

```ts
const needsQuoting = /\s|[^a-zA-Z0-9-]/;
const quoted = needsQuoting.test(family) ? `"${family}"` : family;
```

## UI

Replace the `<EnumSelect>` with a native `<select>` sourced from
the fonts slice. Entries:

1. `System font` (empty value — clears the override).
2. All enumerated system font families, sorted.

The native select is ugly with 2000+ entries but type-to-pick works
for free and it ships fast. Typeahead replacement comes in
Phase 1.5.

## Implementation phases (1)

### 1. System font store

1. Add `src/renderer/store/fontsSlice.ts` with
   `systemFonts: ReadonlyArray<string>` + `loadSystemFonts` action.
   One slice per domain per CLAUDE.md.
2. Call `loadSystemFonts` once at app start (from `App.tsx`).

### 2. Main-process permission grant

Grant `local-fonts` in the main-window `permissionRequestHandler`
for both the dev URL and the packaged `index.html`.

### 3. Fallback helper

1. `src/renderer/lib/fontFallback.ts` — pure functions:
   - `inferFallback(family: string): 'sans-serif' | 'serif' | 'monospace' | 'cursive'`
   - `formatFontValue(family: string): string` — returns
     `"<quoted family>, <fallback>"`.
2. Unit tests in `test/fontFallback.test.ts` covering each branch
   and the quoting edge cases (space, digit, single-word,
   already-quoted input).

### 4. Wire into TypographySection

1. Source options from `useCanvasStore`'s fonts slice (or a
   dedicated store — keep it isolated).
2. Render a native `<select>`.
3. On change: `patchElement(id, { fontFamily: formatFontValue(family) })`.
4. Drop the hardcoded `FONT_FAMILY_OPTIONS` once unused, but keep a
   small `FALLBACK_FONTS` array as the empty-state when
   `queryLocalFonts` is unavailable.

**Acceptance (1):** opening the Typography section on a text
element shows the user's actual installed font families; picking
one writes a quoted family + fallback to the CSS; round-tripping
preserves user hand-edits.

# Phase 1.5 — Typeahead font picker

## Goal

Replace the native `<select>` with a searchable combobox so a
2000-entry font library stays usable.

## UI

- Component: `FontPicker.tsx` (in `components/controls/`).
- Selected value renders as a text field with a dropdown caret.
- Clicking opens a searchable list.
- Typing filters by `family.toLowerCase().startsWith(query)` first,
  then `includes(query)` — prefix matches rank above substring.
- Each row renders in the font itself so the user sees a preview.
- Arrow keys navigate; Enter applies; Escape cancels.
- Windowed rendering: compute visible index range from scroll
  position and row height, render only those rows. No new
  dependency (bundle-size budget per CLAUDE.md).
- Unknown stored values (e.g. an agent-picked family not in the
  enumerated list) render in italic/muted colour rather than
  disappearing.

## Implementation phases (1.5)

### 1. Filter helper

`src/renderer/lib/fontFilter.ts` — pure function that ranks and
filters the font list. Tests in `test/fontFilter.test.ts`: prefix
beats substring, case-insensitive, stable order among ties.

### 2. Windowed list primitive

Either inline inside `FontPicker` (simple — track scrollTop,
offsetTop, row height) or as a `WindowedList.tsx` control if we
reuse the pattern. Start inline, factor out only if needed.

### 3. FontPicker component

Build the component, wire arrow-key / Enter / Escape handling,
render previews in the font itself.

### 4. Swap into TypographySection

Replace the native select with `FontPicker`. Remove `FALLBACK_FONTS`
only if the picker handles the empty-state directly.

# Phase 2 — Google Fonts via embed link

## Data flow

### Where font imports live

Google Fonts imports go at the top of `theme.css`, above the
`:root` block that holds design tokens:

```css
/* scamp: font imports — managed by Project Settings → Fonts */
@import url("https://fonts.googleapis.com/css2?family=Inter:wght@400..700&display=swap");
@import url("https://fonts.googleapis.com/css2?family=Playfair+Display&display=swap");

:root {
  --blue-500: #0000ff;
  /* … */
}
```

Why `theme.css` and not a separate `fonts.css`:

- The user confirmed theme.css is the single per-project
  design-asset file. Colour tokens, font tokens (future), and the
  font imports that back them live together.
- Users who take their project elsewhere only need to import one
  file (`theme.css`) to get their full design system.
- chokidar already watches `theme.css`; Phase 2 doesn't need a new
  file watcher.

### Parsing the embed link

The Google Fonts "Embed" tab gives users one of two snippets:

```html
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400..700&display=swap" rel="stylesheet">
```

or

```html
@import url('https://fonts.googleapis.com/css2?family=Inter:wght@400..700&display=swap');
```

Parser (`src/renderer/lib/googleFontsEmbed.ts`):

1. Accept either form, plus a bare URL.
2. Extract the URL.
3. Validate it's a `fonts.googleapis.com/css2` or `/css` URL.
   Reject arbitrary URLs to keep the attack surface small and
   because we can't reliably parse family names out of arbitrary
   stylesheets.
4. Parse `family=` params and return:
   `{ url: string; families: string[] }`.
5. Handle multi-family URLs (`&family=A&family=B`).

### Extending parseTheme

Today `parseTheme.ts` only extracts `--var` tokens from `:root`.
Extend it to also return the list of `@import` URLs so the fonts
slice can pick them up:

```ts
type ParsedTheme = {
  tokens: ThemeToken[];
  fontImportUrls: string[];
};
```

When we write back, preserve `@import` lines in their original
order. `ThemePanel`'s write path today serializes tokens from
scratch — we'll extend it to emit the imports block first, then
the `:root`. Any non-import, non-`:root` content in `theme.css`
(comments, `@font-face`, media queries) is preserved verbatim.

### Loading fonts on the canvas

`theme.css` isn't imported by the canvas preview today — colour
tokens resolve at paint time via `resolveTokenColor` in the
renderer. For `@import` URLs we need the browser to actually
fetch the stylesheet.

The simplest entry point is the `Viewport` root:

- Effect keyed on `fontImportUrls` that reconciles
  `<link rel="stylesheet" href="<url>">` tags inside the canvas
  document's `<head>`.
- Add a tag per URL on mount; remove on unmount or when the URL
  leaves the list.

Network requests to `fonts.googleapis.com` work out of the box in
Electron's default CSP. If we've tightened CSP elsewhere, add an
explicit allow for `fonts.googleapis.com` and `fonts.gstatic.com`.

### Font picker integration

The fonts slice gains a second source:

```ts
type FontSource = 'system' | 'project';
type AvailableFont = { family: string; source: FontSource };

projectFonts: ReadonlyArray<AvailableFont>;
// derived:
allFonts: ReadonlyArray<AvailableFont>;  // project first, then system
```

Dedupe by lowercase family name, preferring `'project'` so a font
the user actively added beats a same-named local install. The
picker shows a small `Project` badge on project-sourced rows.

## Future work (out of scope here, but on the radar)

- **Font tokens in `theme.css`**. Let users define
  `--headings: "Playfair Display", serif` and
  `--body: "Inter", sans-serif` in the ThemePanel, then pick
  `var(--headings)` from the font picker the same way colour tokens
  work today. Requires:
  - ThemePanel gains a "Font tokens" section alongside colour
    tokens.
  - `parseTheme` classifies tokens by declared type (colour vs font
    vs spacing — probably inferred from value pattern).
  - FontPicker shows tokens as first-class entries above raw
    families.
- **Uploading font files** (`.woff2`, `.ttf`). Users can already
  drop files into `assets/` and hand-write `@font-face` in
  `theme.css` — we just preserve it. A proper UI for that is
  later.
- **Font Squirrel / Adobe / other providers.** Same parser pattern
  as Google Fonts, separate Phase.
- **Variable-font axis controls** (weight as a slider, slant,
  optical size).
- **Per-font custom fallback stacks.** The inference heuristic is
  best-effort; users who want `"Inter", system-ui, -apple-system,
  sans-serif` can hand-edit the CSS.

## Risks

- **`queryLocalFonts` unavailable.** Fall back to the hardcoded
  list on empty / throw; log a warning.
- **Large font lists tanking the picker.** Addressed in Phase 1.5
  by windowing. Phase 1 native `<select>` handles 2000+ entries
  natively even if it looks plain.
- **Family collisions between system and project fonts.** Dedupe by
  lowercase family; prefer `'project'` source.
- **Network-dependent Google Fonts preview.** Offline preview
  falls back to the inferred category. Acceptable — generated code
  is still correct.
- **Agent edits `theme.css`.** If Claude Code writes a raw
  `@font-face` block, the parser should preserve it verbatim even
  though we don't turn it into a token or an import.
- **Artboard-background migration.** Old projects have the colour
  on the global `Settings` object. The Phase 0.5 handler writes
  the default (`#0f0f0f`) into the new `scamp.config.json` on
  first open — users who had customized the global value will see
  it revert to default inside their projects. Acceptable given
  the feature is only weeks old and the value is trivially
  re-picked. Note in release notes if we cut a release between
  these phases.
- **CSP blocking Google Fonts.** If tightened CSP is in play,
  whitelist `fonts.googleapis.com` + `fonts.gstatic.com` during
  Phase 2.
- **ThemePanel write path stomps imports.** The current serializer
  rewrites the whole file. Phase 2 must preserve the imports block
  and any non-token content between imports and `:root`. Unit
  tests in `parseTheme.test.ts` cover this explicitly.
