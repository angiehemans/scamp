# Nudge with arrow keys — Plan

**Status:** Implemented (both phases complete).
**Date:** 2026-04-14
**Backlog item:** #10

## Goal

Match Figma-style keyboard nudging in two places:

1. **Canvas** — selected element moves 1px with arrow keys, 10px with
   Shift+arrows.
2. **Number inputs in the properties panel** — up/down arrow change the
   value by 1, Shift+up/down by 10.

---

## Current state

**Canvas:** No arrow-key handling for selected elements exists. The
global keyboard listener in `ProjectShell.tsx` already catches window
keydown events for copy/paste/delete/undo and ignores them when focus
is on an editable target via `isEditableTarget`. Arrow keys are a
natural fit for the same listener.

The store exposes `moveElement(id, x, y)` — it's already used by drag
interactions and sets both coordinates in one update. Bounds clamping
lives in the interaction layer (`clampToParent`), not in the store, so
we'll duplicate that clamp logic at the nudge call site OR lift it.

The existing drag flow also skips flex children because their position
is owned by the flex layout engine. Nudge needs the same guard
(`isFlexChild`) and should skip the page root (which can't move).

**Number inputs:** `NumberInput` is the single entry point for every
number field in the WYSIWYG panel (SizeSection, PositionSection,
LayoutSection, TypographySection). It uses `type="text"` rather than
`type="number"` so browser-default arrow-key stepping doesn't apply —
which is what we want, because we're about to implement our own.

Important existing behavior to preserve:

- The component keeps a local `draft` string so users can type
  intermediate values. It calls `onChange` only on explicit commit
  (blur or Enter).
- `min` / `max` props exist and are enforced in `commit`.
- When `value` is `undefined` and `allowEmpty` is true, the input shows
  empty.

## Scope decisions

### Multi-select

Backlog language is singular ("a selected element") but Delete and
Duplicate both operate on the full selection today, and Figma nudges the
whole selection. I'll apply nudge to every selected element that passes
the eligibility check (not root, not flex child). Consistent with the
rest of the app.

### Clamping

Canvas nudge clamps to the element's parent bounds, matching the drag
behavior. I'll reuse the same `clampToParent` helper by lifting it into
a small shared module (`src/renderer/lib/bounds.ts`) so both the
interaction layer and the nudge handler import it.

Number input clamping already exists via `min` / `max` on the field.
Arrow keys respect the same bounds — hitting ⬆ at `max` does nothing,
hitting ⬇ at `min` does nothing.

### File-write flooding

The story calls out "not on every individual keydown, to avoid flooding
the file system with writes while the key is held". Two things to
verify during implementation:

- The sync bridge already debounces disk writes on state changes; if
  that debounce is sufficient (say 100–200ms) we don't need extra
  throttling at the nudge site. Nudging fires state updates, the
  debounce handles the write frequency.
- If the existing debounce is too short or nonexistent, add a small
  `nudgeInFlight` flag or throttle in the handler.

I'll check the sync bridge during Phase 1 and adjust if needed.

### Auto-repeat (holding the key down)

Browsers fire `keydown` repeatedly while an arrow key is held. I want
this — it matches Figma and is what "the user should see the canvas
respond in real time as they hold the key" means. No special handling
needed; the handler fires on each repeat.

### `undefined` values in number inputs

If the field currently has `value === undefined` (an unset font-size,
line-height, etc.), arrow keys treat it as `0` for the first press
(or `min` if that's defined and higher). After the first press the
value is concrete and subsequent arrows work normally. Simpler than
inventing a "last seen" fallback and gives the user a fast path out
of `undefined`.

### Canvas focus

The `ProjectShell` keydown listener is attached to `window`, not the
canvas. We don't need `tabIndex` gymnastics — the existing
`isEditableTarget` check covers the "focus is in an input" case. The
listener fires globally and `document.activeElement` stays consistent
with what the user is interacting with.

## Implementation phases

### Phase 1 — Canvas nudge

1. Extract `clampToParent` from
   `src/renderer/src/canvas/CanvasInteractionLayer.tsx` into
   `src/renderer/lib/bounds.ts` (pure function, same signature). Update
   the interaction layer import.
2. In `ProjectShell.tsx`, extend the existing keydown handler to match
   arrow keys:
   - Bail on `isEditableTarget(e.target)`.
   - Bail if `selectedElementIds` is empty OR the only id is root.
   - Determine `(dx, dy)` from the key (Arrow{Up,Down,Left,Right}) and
     multiply by 10 if `e.shiftKey`.
   - For each selected id:
     - Skip root.
     - Look up the element; skip when its parent is a flex container
       (flex layout owns position).
     - Compute the proposed `(x, y)`, clamp against the parent's
       bounds via `clampToParent`.
     - Call `moveElement(id, newX, newY)`.
   - `preventDefault` so the browser doesn't scroll the page while
     nudging.
3. Verify the sync bridge's debounce is sensible when arrow keys
   auto-repeat at ~30 Hz. If not, add throttling at the nudge site.
4. Add unit test for a tiny `computeNudgeDelta(key, shift)` helper if
   the logic grows — otherwise inline and skip.

**Acceptance:**

- Selecting a rectangle and pressing → moves it 1px right; Shift+→
  moves it 10px.
- Selecting nothing and pressing → does nothing.
- Selecting root and pressing → does nothing (root doesn't move).
- Selecting a flex child and pressing → does nothing; the element
  stays in its flex flow.
- Focus on a property-panel input and pressing → adjusts the input
  (Phase 2), NOT the canvas element.
- Pressing → near the right edge clamps at the parent boundary.
- File on disk updates after the nudge stops (exact same pathway as a
  drag).

### Phase 2 — Number input nudge

1. In `NumberInput`, extend `onKeyDown`:
   - `ArrowUp` → step = +1; Shift+ArrowUp → +10.
   - `ArrowDown` → step = -1; Shift+ArrowDown → -10.
2. Compute the next value from the current `draft` (fall back to
   `value`, then to `0` or `min` if both are absent).
3. Clamp against `min` / `max`.
4. Update `draft` and call `onChange(next)` immediately so the canvas
   reflects the change in real time.
5. `preventDefault` so the cursor doesn't jump to the input's start/end
   (default `type="text"` behavior).
6. Leave `Enter` / `blur` commit behavior alone.

**Acceptance:**

- Focus a width input with value 100, press ⬆ → shows 101 and the
  rectangle grows by 1px.
- Hold ⬆ → the value and canvas animate upward continuously.
- Shift+⬆ steps by 10.
- ⬇ on a width at 0 (min=0) does nothing.
- ⬆ on a font-size that's `undefined` sets it to 1 (or the field's
  `min` if higher) and the canvas reflects it.
- ⬆ inside the `PageNameInput`, any text input, or the CSS editor is
  unaffected (only `NumberInput` has the new handler; other inputs
  keep their native behavior).

## Out of scope

- Nudging grouped selections with different parents (we'll skip each
  element that can't nudge in its parent context; we won't try to
  "move the selection as a rigid unit").
- Modifier for arbitrary step sizes (e.g. Ctrl+arrow for 100px) —
  sticking to the Figma-standard 1 / 10 split.
- Keyboard resize (Shift+Option+arrows in Figma) — that's a separate
  interaction.
- Nudging in the layers panel — layers panel is for hierarchy, not
  spatial position.
- Any work on `PositionSection`, `SizeSection` layout — they already
  use `NumberInput` so they inherit the new behavior for free.

---

## Risks

- **Global keydown + third-party editors.** The CodeMirror editors
  (CSS panel, code panel) capture keyboard events via `contenteditable`
  elements that are *inside* our pass-through focus check. The existing
  `isEditableTarget` check already returns true for them, so arrow keys
  there won't bubble into our nudge handler. Worth smoke-testing
  because CodeMirror is idiosyncratic.

- **Terminal.** xterm.js in `TerminalPanel` captures keys directly —
  but only when the pty view has focus. Same as CodeMirror, the focus
  check protects us.

- **Flex child selected + multi-select mix.** If the user selects a
  rectangle (movable) and a flex child (not movable), the rectangle
  should nudge and the flex child should stay put. This falls out of
  the per-element skip in the handler. Documented in acceptance.

- **File write frequency.** Holding an arrow key fires ~30 keydown
  events per second. If the sync bridge writes on every state change
  with no debounce, that's 30 file writes per second. Pre-existing
  drag interactions have the same shape (~60 pointermoves per second)
  so I expect the debounce is already tuned, but I'll verify during
  Phase 1.
