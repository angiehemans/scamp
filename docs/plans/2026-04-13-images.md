# Images — Plan

**Status:** Implemented (all 5 phases complete).
**Date:** 2026-04-13

## Goal

Two ways to use images in Scamp:

1. **Image element** — place an `<img>` tag on the canvas via the toolbar
   (`I`) or by dragging an image file from the OS onto the canvas.
2. **Background image** — set `background-image` on any rectangle from
   the properties panel, with controls for size, position, and repeat.

Both copy the image file into the project's `assets/` folder and
reference it via a relative path.

---

## Data model changes

### New element type: `image`

Add `'image'` to the `ScampElement.type` union and two new fields:

```ts
type: 'rectangle' | 'text' | 'image';
src?: string;   // relative path, e.g. "../assets/hero.png"
alt?: string;   // accessibility text
```

These fields are only meaningful when `type === 'image'`. Rectangle and
text elements ignore them.

### Background image on rectangles

No new fields on ScampElement. Background images flow through the
existing `customProperties` bag:

```css
background-image: url("../assets/hero.png");
background-size: cover;
background-position: center;
background-repeat: no-repeat;
```

The properties panel writes these as CSS declarations via `patchElement`
on `customProperties`. They round-trip through `generateCode` /
`parseCode` unchanged (already works today for any custom CSS). The
panel just provides a nice UI for editing them.

## IPC

### New channel: `file:copyImage`

```ts
type CopyImageArgs = {
  sourcePath: string;   // absolute path to the source file
  projectPath: string;  // the project root folder
};

type CopyImageResult = {
  relativePath: string;  // e.g. "../assets/hero.png"
  fileName: string;      // e.g. "hero.png"
};
```

The main process handler:
1. Creates `assets/` in the project root if it doesn't exist.
2. Copies the file (deduplicates by appending a suffix if the name
   already exists: `hero.png` → `hero-1.png`).
3. Returns the relative path for use in `src` or `background-image`.

Lives in a new file `src/main/ipc/image.ts`.

### New channel: `file:chooseImage`

Opens a native file dialog filtered to image types (PNG, JPG, WebP,
SVG, GIF). Returns the selected file path or null if cancelled. Used by
the "Set background image" button in the properties panel.

## Implementation phases

### Phase 1 — Asset infrastructure

1. Add `CopyImageArgs`, `CopyImageResult` to `src/shared/types.ts`.
2. Add `FileCopyImage`, `FileChooseImage` to `ipcChannels.ts`.
3. Create `src/main/ipc/image.ts` with:
   - `copyImage` — creates `assets/`, copies file, returns relative path.
   - `chooseImage` — native file dialog for images.
4. Register in `src/main/index.ts`.
5. Expose in `src/preload/index.ts`.
6. Create `assets/` on project creation (empty folder, or skip and
   create on first use).
7. Update `agent.md` to document the `assets/` folder.

**Acceptance:** IPC channels work, images can be copied into projects.

### Phase 2 — Image element type

1. Add `'image'` to `ScampElement.type`, add `src?: string`,
   `alt?: string`.
2. Add `'image'` to the `Tool` union in `canvasSlice.ts`.
3. Add `createImage` action to the store (similar to `createText` but
   creates an image element with the given src).
4. Update `classNameFor` — image elements use `img_` prefix.
5. Update `defaultTagFor` — image → `'img'`.
6. Update `renderJsx` in `generateCode.ts`:
   - Image elements emit `<img data-scamp-id="..." className={...} src="..." alt="..." />`
   - Always self-closing (no children, no text).
7. Update `inferElementType` in `parseCode.ts` — `img_` prefix → `'image'`.
8. Update `parseTsxStructure` — extract `src` and `alt` attributes.
9. Update `makeBaseline` — carry `src` and `alt` through.
10. Update `ElementRenderer.tsx` — render `<img>` tags with `src` and
    proper sizing styles.
11. Update `elementToStyle` — image elements use `object-fit: cover`
    by default, and `width`/`height` from the element's sizing fields.

**Acceptance:** image elements generate correct TSX/CSS, round-trip
through parseCode, and render on the canvas.

### Phase 3 — Image tool + drag-and-drop

1. Add image tool to the Toolbar (`I` shortcut, `IconPhoto` icon).
2. Add keyboard shortcut `I` in Toolbar's keydown handler.
3. When the image tool is active and the user clicks the canvas:
   - Open a native file dialog via `file:chooseImage`.
   - Copy the file via `file:copyImage`.
   - Call `createImage` with the returned relative path.
   - Switch back to select tool.
4. Add drag-and-drop on `CanvasInteractionLayer`:
   - `onDragOver` — check for image files in `e.dataTransfer.types`,
     prevent default.
   - `onDrop` — read the dropped file path, copy via IPC, create an
     image element at the drop position.
   - Works regardless of which tool is active.
5. Set the cursor to `crosshair` when the image tool is active (same
   as rectangle tool).

**Acceptance:** user can place images via toolbar click or file drag.

### Phase 4 — Background image controls

1. In `BackgroundSection.tsx`, add a "Set background image" button
   below the color picker.
2. Clicking it opens the native file dialog, copies the file, then
   writes the four background properties to `customProperties`:
   ```
   background-image: url("../assets/file.png")
   background-size: cover
   background-position: center
   background-repeat: no-repeat
   ```
3. When the element has a `background-image` in `customProperties`,
   expand the section to show:
   - **Background size**: segmented control (Cover / Contain / Auto).
   - **Background position**: segmented 9-point grid (top-left,
     top-center, top-right, center-left, center, center-right,
     bottom-left, bottom-center, bottom-right).
   - **Background repeat**: segmented (No repeat / Repeat / Repeat X /
     Repeat Y).
   - **Remove background image** button — clears all four properties.
4. Each control writes to `customProperties` via `patchElement`.
5. The canvas already renders `customProperties` via the spread in
   `elementToStyle`, so background images show up automatically.

**Acceptance:** user can set/edit/remove background images on any
rectangle with a visual UI.

### Phase 5 — Image-specific properties panel

1. Create `ImageSection.tsx` — shown when the selected element is type
   `image`. Controls:
   - **Source**: read-only path display + "Replace" button (opens file
     dialog, copies new file, updates `src`).
   - **Alt text**: text input.
   - **Object fit**: segmented (Cover / Contain / Fill / None).
   - **Object position**: same 9-point grid as background-position.
2. Wire into `UiPanel.tsx` — show ImageSection when `el.type === 'image'`.
3. Hide the Background section for image elements (background-color
   doesn't apply to `<img>`).

**Acceptance:** image elements have a dedicated properties section.

## Canvas rendering

Image elements render as real `<img>` tags in the DOM via
`ElementRenderer`. The `src` attribute uses the relative path which
resolves against the renderer's base URL. For dev mode, electron-vite
serves the project folder, so relative paths work. For production,
the path resolves relative to the HTML file — may need a `file://`
protocol prefix.

Background images on rectangles already work via `customProperties` →
inline style spread. No special rendering logic needed.

## Out of scope

- Image cropping / masking
- Image optimization / compression
- Drag-to-resize images while maintaining aspect ratio (uses the same
  resize handles as rectangles — no aspect lock for now)
- Uploading images to a CDN
- SVG inline embedding (SVGs are treated as raster images via `<img>`)
- Video or animated content

---

## Risks

- **Relative path resolution.** In dev mode, electron-vite serves
  files from the project folder so `../assets/hero.png` resolves
  correctly. In production, `<img src="../assets/hero.png">` resolves
  relative to the Electron renderer's HTML file, NOT the project
  folder. The `ElementRenderer` may need to resolve `src` to an
  absolute `file://` path for canvas preview. The generated TSX keeps
  the relative path (correct for the user's build system).

- **Large images.** No size limits or optimization. A user could drop
  a 50MB PNG. The canvas renders it at whatever size the element is
  set to, so performance depends on the browser's image handling.
  Acceptable for POC.

- **File name collisions.** The copy handler deduplicates by appending
  a numeric suffix (`hero.png` → `hero-1.png`). Simple and reliable.
