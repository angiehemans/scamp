# Add, Duplicate, and Delete Page — Plan

**Status:** Implemented (all 6 phases complete).
**Date:** 2026-04-14
**Backlog item:** #9

## Goal

Let users manage pages entirely from the Pages sidebar — create new
ones, duplicate existing ones, and delete ones they no longer need —
without leaving the app. All three flows operate on real files in the
project folder.

---

## Current state

Backend is partly in place:

- `IPC.PageCreate` exists in `src/main/ipc/page.ts` with name
  validation (`^[a-zA-Z0-9-]+$`) and writes `[name].tsx` +
  `[name].module.css` using the standard empty-page template.
- `IPC.PageDelete` exists and removes both files.
- `preload/index.ts` exposes `createPage` and `deletePage`.

Gaps:

- **No UI** — the sidebar only lists existing pages, no "+ Add Page"
  button, no right-click menu, no way to delete.
- **No duplicate IPC** — would have to be added.
- **`PageCreate` silently overwrites** an existing file. User story says
  names must be unique, so this needs an existence check.
- **`PageDelete` has no guardrails** — it'll happily delete the last
  page, leaving the project with no pages to open. Need a minimum-of-one
  check in the UI (main handler stays agnostic).
- **`App` state** owns `project` and therefore `project.pages`. Adding
  or removing a page needs a way to refresh that list. Today there's no
  callback from `ProjectShell` up to `App` for project changes.

---

## Data flow

### New IPC: `page:duplicate`

```ts
type PageDuplicateArgs = {
  projectPath: string;
  sourcePageName: string;   // e.g. "home"
  newPageName: string;      // e.g. "home-copy"
};
// Returns PageFile for the new page.
```

Handler steps:

1. Validate `newPageName` with the existing `PAGE_NAME_RE`.
2. Ensure `newPageName` doesn't already exist on disk (both `.tsx` and
   `.module.css`).
3. Read `[sourcePageName].tsx` and `[sourcePageName].module.css`.
4. In the TSX:
   - Replace the `import styles from './[source].module.css'` line to
     point at the new module.
   - Replace `export default function [OldComponent]()` with the new
     component name derived from `newPageName` (same
     `componentNameFromPage` helper).
5. Write both files.
6. Return a `PageFile`.

### Update: `page:create`

Add a pre-write existence check. If either target file already exists,
throw with a clear message (`A page named "..." already exists`). Keep
validation regex the same.

### Existing: `page:delete`

Handler stays as-is (it already removes both files by name). The UI
layer owns all policy:

- Refuse to delete the last remaining page — the project must always
  have at least one page so the canvas has something to render.
- Prompt for confirmation before deleting.
- If the deleted page is the active page, pick a sibling to switch to
  (the first remaining page in the list).

### Project state refresh

`ProjectShell` gets a new optional prop:

```ts
onProjectChange?: (next: ProjectData) => void;
```

`App.tsx` passes `setProject`. After a successful create/duplicate the
shell builds a new `ProjectData` with the added page appended to
`pages`; after a successful delete it builds a new one with the page
removed from `pages`. Either way it calls `onProjectChange` and, when
appropriate, switches `activePageName`.

No extra file watching needed — we already have the file contents from
the IPC return value, so we can splice them in directly. The existing
chokidar watcher will fire for the affected files but the sync bridge
already tolerates "file changed for current page" and no-ops when
content matches in-memory state.

## UI

### Shared: `PageNameInput` component

`src/renderer/src/components/PageNameInput.tsx` — inline text input used
by both flows.

Props:

```ts
type Props = {
  initialValue?: string;
  /** Names already taken — for inline error feedback. */
  existingNames: ReadonlyArray<string>;
  onConfirm: (name: string) => void;
  onCancel: () => void;
};
```

Behaviour:

- Autofocuses on mount; selects existing text so a duplicate name can be
  edited in place.
- On each keystroke: validate against `PAGE_NAME_RE` and `existingNames`.
  Show a small red hint under the input when invalid.
- Enter → confirm (only when valid).
- Escape → cancel.
- Blur → cancel (not confirm) so the user doesn't accidentally create a
  file by clicking away mid-typing. This mirrors how the layers-panel
  rename works.

Styling matches the existing inline rename input in `ElementTree`.

### "+ Add Page" button

Rendered at the bottom of the Pages sidebar section in `ProjectShell`.
Thin full-width button (`+ Add Page`). Clicking swaps it for the
`PageNameInput` with an empty value. Confirm calls
`window.scamp.createPage`, then `onProjectChange` + `setActivePageName`.

Show a brief disabled/loading state on the input while the IPC is in
flight (button text becomes "Creating…"). If the IPC rejects, surface
the error under the input and leave it open so the user can fix the
name.

### Right-click menu on page rows

Each `<button>` in `project.pages.map` gets `onContextMenu` that
positions a small floating menu at the cursor. Menu items:

- **Duplicate**
- **Delete** (destructive — styled red; disabled when there's only one
  page in the project)

`src/renderer/src/components/PageContextMenu.tsx` — bare-bones menu.
Closes on outside click, Escape, or after an item is chosen. Ordinary
left-click on a page row keeps its current selection behaviour.

**Duplicate** turns the target row into the `PageNameInput` seeded with
`[name]-copy`, selects the "-copy" portion so the user can immediately
retype it. Confirm calls `window.scamp.duplicatePage`.

**Delete** opens a small confirmation dialog (`ConfirmDialog` — a
generic modal we can reuse later). Confirming calls
`window.scamp.deletePage`, then updates `project.pages` via
`onProjectChange`. If the deleted page was the active page, the shell
switches to the first remaining page. Cancel closes the dialog with no
side effects.

Future-proofing: the menu is built as a reusable list so we can add
Rename later without reworking the mount logic. Rename is explicitly
out of scope for this story (see backlog #11).

### Sidebar tweaks

- Add a subtle hover highlight to page rows so they feel
  right-clickable.
- The Add button sits below the list, not inside it.
- When either flow is editing an inline name, disable page switching
  (clicks on other rows do nothing) so the user doesn't lose their typed
  name by clicking away.

## Implementation phases

### Phase 1 — Backend

1. Add `PageDuplicateArgs` to `src/shared/types.ts`.
2. Add `IPC.PageDuplicate` to `src/shared/ipcChannels.ts`.
3. In `src/main/ipc/page.ts`:
   - Add `handleDuplicate`.
   - Tighten `handleCreate` to reject on existing files.
   - Register `IPC.PageDuplicate`.
4. Expose `duplicatePage` in `src/preload/index.ts`.

**Acceptance:** from devtools, calling `window.scamp.createPage` or
`duplicatePage` writes the expected files and throws on collisions.

### Phase 2 — `ProjectShell` plumbing

1. Add `onProjectChange` prop to `ProjectShell`.
2. Wire `setProject` through from `App.tsx`.
3. Introduce `editingPage` state: `'new' | { duplicate: string } | null`
   — drives whether an inline input is shown and where.

**Acceptance:** component compiles; no visual change yet.

### Phase 3 — `PageNameInput`

1. Build the component and CSS module.
2. Unit tests for the name validator helper only (pure function):
   - Rejects spaces, uppercase, empty.
   - Rejects names in `existingNames`.
   - Accepts valid names.

**Acceptance:** tests pass; component renders in a Storybook-less
harness manually.

### Phase 4 — Add Page UI

1. Render the "+ Add Page" button in the sidebar.
2. Wire click → show `PageNameInput`.
3. Confirm → call IPC → on success update `project` via
   `onProjectChange` + switch `activePageName`.
4. Error surface under the input on IPC rejection.

**Acceptance:** clicking "+ Add Page", typing `about`, pressing Enter
creates `about.tsx` + `about.module.css` and switches to the new page.

### Phase 5 — Context menu + Duplicate Page UI

1. Build `PageContextMenu`.
2. Wire `onContextMenu` on page rows → open menu at cursor.
3. "Duplicate" → show `PageNameInput` in place of the target row, seeded
   with `[name]-copy`.
4. Confirm → call `duplicatePage` IPC → on success update `project` and
   switch to new page.

**Acceptance:** right-clicking `home`, choosing Duplicate, pressing
Enter creates `home-copy.tsx` with the component renamed to
`HomeCopy`, CSS copied verbatim with its import path updated.

### Phase 6 — Delete Page UI

1. Build `ConfirmDialog` — a small generic modal with a title, body,
   and Cancel/Confirm buttons. The confirm button variant is
   configurable (`'primary' | 'destructive'`).
2. Add a "Delete" item to `PageContextMenu`. Disable it when
   `project.pages.length === 1`.
3. Clicking Delete opens the dialog: *"Delete page `about`? This will
   remove `about.tsx` and `about.module.css` from your project folder.
   This cannot be undone."*
4. Confirm → call `deletePage` IPC → on success:
   - Remove the page from `project.pages` via `onProjectChange`.
   - If the deleted page was the active one, switch to the first
     remaining page.

**Acceptance:** right-clicking `about`, choosing Delete, confirming
the dialog removes both files from disk and from the sidebar. Delete
option is disabled when only one page remains.

## Out of scope

- Rename an existing page — separate story (backlog #11). Renaming has
  the same atomic two-file rewrite problem the element rename solved,
  plus it needs to update the component name and module import inside
  the TSX. Big enough to stand alone.
- Undo for delete — deleting writes to disk immediately and isn't
  tracked in the undo stack. The confirmation dialog is the only guard.
- Reordering pages — the page list order is "whatever `readdir` returns"
  today, which is fine for now.
- Any kind of page templates beyond the default empty page.
- Page folders / nesting.

---

## Risks

- **Component-name rewrite on duplicate.** The regex for finding
  `export default function [Name]()` has to match the exact format
  emitted by `defaultPageTsx` AND survive manual edits the user might
  have made (adding props, extra whitespace, etc.). Plan: use a tight
  regex that matches `export default function \w+\s*\(\s*\)` and only
  replace the name capture group. If the match fails, fall back to
  leaving the component name as-is — the file still works, it just
  keeps the old component name. Log a warning for now; we can add a
  proper error surface later if it bites.

- **chokidar double-fires.** Writing two new files will fire the
  watcher. The sync bridge currently handles changes to the *active*
  page; new files that aren't the active page should be ignored but
  also, ideally, picked up as "there's a new page available". Today
  that second case doesn't happen because nothing else writes new
  `.tsx` files. This plan sidesteps the issue by returning the new
  `PageFile` from the IPC and updating `project.pages` directly, so we
  don't depend on the watcher to learn about the new page. If the
  watcher *does* fire, the list will already contain the page and the
  update is a no-op.

- **Focus management for the inline input.** The sidebar contains
  buttons that fight for keyboard focus. The input needs explicit
  `autoFocus` + `select()` on mount, and Escape should re-focus the
  sidebar container so subsequent shortcuts still work.

- **Name collision check is racy.** Another process could create the
  file between our `fs.access` check and our `fs.writeFile`. Acceptable
  for a local single-user tool; we'll still catch the write error and
  surface it.
