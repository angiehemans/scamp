# Export Pages and Elements — Plan

**Status:** Draft for review.
**Date:** 2026-05-06
**Source:** `docs/backlog-4.md` story #3
**Related:** Box shadow / blend modes (just shipped — same IPC-channel
constant pattern). Background image upload in `image.ts` is the
closest existing example of "renderer asks the main process to write
a file the user picks via a native dialog."

---

## Goal

Let users export the canvas (or a single element with its children)
as PNG, SVG, or PDF without leaving the app. PNG and SVG capture
happens in the renderer (`html-to-image`); PDF goes through
Electron's `webContents.printToPDF()` via a hidden offscreen window
in the main process. All three end up in a native save dialog so
the user picks where the file lands.

The export reflects the *current canvas state* — not the last saved
file state — so a user mid-edit gets what they see on screen.

The export controls live as a permanent section at the bottom of
the WYSIWYG panel (Figma-style), not as a centered modal. The
section's *scope* is implicit: when an element is selected, it
exports that element; otherwise it exports the page. PDFs are
always canvas-sized — no page-size or orientation picker, since
the user has already designed at a known canvas size.

---

## Current state — what we can build on

- **IPC channel pattern** (`src/shared/ipcChannels.ts`,
  `src/main/ipc/<domain>.ts`, `src/preload/index.ts`). The
  background-image upload (`FileChooseImage` / `FileCopyImage`
  in `image.ts`) is a near-identical case: renderer triggers a
  flow, main process pops a native dialog, writes a file, returns
  a result.
- **`PageContextMenu`** (`src/renderer/src/components/PageContextMenu.tsx`)
  is a generic right-click menu primitive — already supports
  outside-click and Escape dismissal. The new element context menu
  reuses this component verbatim.
- **Canvas DOM markers** — every rendered element carries
  `data-element-id="<id>"` and `data-scamp-id="<className>"`
  (`ElementRenderer.tsx:495`). The capture code locates the right
  DOM node via `[data-element-id="…"]` for element-scoped exports.
- **Frame DOM marker** —
  `Viewport.tsx:154` renders the canvas with
  `data-testid="canvas-frame"` and `data-canvas-width` /
  `data-canvas-scale` attributes. We can reuse the `data-testid`
  selector OR add a dedicated `data-canvas-export-root` for
  clarity. The frame has `transform: scale(zoom)` on it, which we
  must reset before capture so the image is at the design's
  intrinsic size, not the user's current zoom.
- **`PropertiesPanel` / `UiPanel`** layout
  (`src/renderer/src/components/PropertiesPanel.tsx`,
  `UiPanel.tsx`). The new Export section drops in next to the
  existing element-level sections (Visibility, Transitions,
  Animation) at the bottom of the panel. The panel already
  renders even when nothing is selected (showing the keyboard
  shortcuts table) — we'll restructure that empty state to
  also render the page-scoped Export section.
- **Global keyboard handler** (`ProjectShell.tsx`) — already wires
  Cmd+C / Cmd+V / Cmd+D / Cmd+Z / Cmd+G / etc. The
  `Cmd+Shift+E` shortcut adds one more branch.
- **Project context** — the canvas store exposes
  `s.projectPath` (the absolute project folder), used as the
  default-save-dir hint per the story spec.

What's NOT there yet:

- No `html-to-image` dep — needs to be added (`package.json`).
- No element-level right-click context menu (only the per-page one
  in the layers panel).
- No save-dialog IPC — `dialog.showOpenDialog` is used for image
  uploads; we need `dialog.showSaveDialog` paired with a
  binary-file write.
- Canvas chrome (selection outlines, resize handles, interaction
  layer) currently sits as siblings of `ElementRenderer` inside
  the frame, with no DOM marker to distinguish "design content"
  from "editor chrome." `html-to-image`'s `filter` callback needs
  a deterministic way to skip chrome nodes.

## Non-goals for this story

- **Multi-element export.** The story is explicit: one element or
  the full page. Selecting two elements and exporting picks the
  first selection (or treats it as "export full page" — see open
  questions).
- **State / variant capture.** Exports the default rest state.
  Hover, active, focus, animations mid-flight are not captured
  even if the canvas is currently previewing them. Animations
  pause at their current frame for the capture to keep the
  interaction predictable.
- **Server-side rendering or CLI export.** This is a renderer +
  Electron-main flow. No Node-side HTML rendering.
- **Image-format options beyond PNG/SVG/PDF** (JPEG, WebP, AVIF).
  Future work; PNG covers the "share with stakeholders" use case.
- **Custom watermarks / branding.** Out of scope.
- **Persistent settings storage.** "Remembers last format and
  scale per session" — in-memory state only, lost on app close.
- **PDF page size and orientation.** Removed from the v1 panel.
  PDFs are always canvas-sized at the canvas's current portrait
  orientation. Agents who want a different paper size can still
  print the exported PDF themselves. A future story can add the
  picker back if users ask for it.

## IPC contract

### Channels

```ts
// src/shared/ipcChannels.ts — new entries

ExportPng: 'export:png',
ExportSvg: 'export:svg',
ExportPdf: 'export:pdf',
ExportChooseSavePath: 'export:chooseSavePath',
```

Splitting the dialog from the file write keeps each handler small
and testable. The renderer:

1. Calls `chooseSavePath({ filename, format })` → returns
   `{ canceled: boolean; path: string | null }`.
2. If not canceled, captures the image / SVG in the renderer (or
   the renderer asks main to do it for PDF), then calls
   `exportPng` / `exportSvg` / `exportPdf` with the path and
   payload.

This split also lets us short-circuit the expensive capture step if
the user cancels the save dialog.

### Payload types

```ts
// src/shared/types.ts — new types

export type ExportFormat = 'png' | 'svg' | 'pdf';

export type ExportChooseSavePathArgs = {
  filename: string;        // suggested default, no extension
  format: ExportFormat;
  /** Default folder hint — typically the project path. */
  defaultDir?: string;
};

export type ExportChooseSavePathResult = {
  canceled: boolean;
  path: string | null;
};

export type ExportPngArgs = {
  /** Captured PNG as a base64 data URL (e.g. `data:image/png;base64,…`). */
  dataUrl: string;
  path: string;
};

export type ExportSvgArgs = {
  svgString: string;
  path: string;
};

export type ExportPdfArgs = {
  /**
   * Standalone HTML payload to print. The renderer assembles a
   * print-ready document (the element/page + all required
   * `<style>` tags + the project's theme tokens) and hands it to
   * main as a string. Main loads it into a hidden offscreen
   * BrowserWindow and calls webContents.printToPDF(). Decoupling
   * the print HTML from the live canvas avoids capturing editor
   * chrome and lets us print at intrinsic size regardless of
   * canvas zoom.
   */
  html: string;
  path: string;
  /**
   * Canvas dimensions in CSS pixels. The PDF page is sized to
   * exactly these — no paper-size or orientation picker. For
   * element exports this is the element's bounding box; for page
   * exports it's the canvas frame's intrinsic dimensions.
   */
  widthPx: number;
  heightPx: number;
};

export type ExportResult = {
  ok: boolean;
  /** Populated on failure — short, user-readable. */
  error?: string;
};
```

### Preload surface

Three new entries in `src/preload/index.ts`:

```ts
exportChooseSavePath: (args: ExportChooseSavePathArgs):
  Promise<ExportChooseSavePathResult> =>
  ipcRenderer.invoke(IPC.ExportChooseSavePath, args),

exportPng: (args: ExportPngArgs): Promise<ExportResult> =>
  ipcRenderer.invoke(IPC.ExportPng, args),

exportSvg: (args: ExportSvgArgs): Promise<ExportResult> =>
  ipcRenderer.invoke(IPC.ExportSvg, args),

exportPdf: (args: ExportPdfArgs): Promise<ExportResult> =>
  ipcRenderer.invoke(IPC.ExportPdf, args),
```

### Main-process handler — `src/main/ipc/export.ts`

A single module mirroring `image.ts`'s shape. Exports a
`registerExportHandlers(win: BrowserWindow)` that the main
`index.ts` boot calls once. The four handlers:

- **`export:chooseSavePath`** — `dialog.showSaveDialog(win, {…})`
  with the right default extension and filter list per format.
- **`export:png`** — strip the data-URL prefix, decode base64,
  write to disk via `fs/promises.writeFile`. Return `{ ok: true }`
  or `{ ok: false, error }`.
- **`export:svg`** — write the string verbatim with utf-8
  encoding.
- **`export:pdf`** — open an offscreen `BrowserWindow` (no
  visible window, `show: false`, `width / height` matching the
  supplied `widthPx` / `heightPx` in CSS pixels),
  `loadURL('data:text/html;charset=utf-8,…')` with the supplied
  HTML, `webContents.once('did-finish-load', …)`, call
  `printToPDF()` with `pageSize: { width, height }` derived from
  `widthPx` / `heightPx` (converted to microns:
  `px * 26.4583`), `landscape: false`, `printBackground: true`.
  Write the buffer, destroy the window. Errors at any step return
  `{ ok: false, error }`.

### Sandbox / safety

- **Path validation.** The save path comes from a user-driven
  native dialog, so it's already authoritative. We don't need to
  validate the path against the project root — the user might
  legitimately export to Downloads, Desktop, etc.
- **Dialog parent.** All dialogs use the main `BrowserWindow` as
  parent so they're modal to the app, not floating loose.
- **Hidden PDF window.** The offscreen `BrowserWindow` is opened
  with `webPreferences: { sandbox: true, nodeIntegration: false,
  contextIsolation: true }` — same as the main renderer. The PDF
  render context never accesses the project files; the HTML
  payload is fully self-contained.

## Renderer architecture

### State — `canvasSlice.ts`

Add session-only export-panel state to the canvas store:

```ts
type ExportScope =
  | { kind: 'page' }
  | { kind: 'element'; elementId: string };

type Slice = {
  // ... existing ...
  exportSettings: {
    /** Persisted across the session. */
    lastFormat: ExportFormat;
    lastPngScale: 1 | 2 | 3;
  };
  setExportFormat: (next: ExportFormat) => void;
  setExportPngScale: (next: 1 | 2 | 3) => void;
};
```

Initial defaults: `lastFormat: 'png'`, `lastPngScale: 2`.

No `open` / `scope` state — the section is always visible, and
its scope is derived implicitly from the selection
(`s.selectedElementIds[0]` exists → element scope, else page
scope) at export time. The session-only persistence is just the
two settings the user might tweak between exports.

### Scope derivation

The Export section is always mounted; the scope it operates on is
implicit:

- `selectedElementIds.length === 0` → page scope (capture the
  canvas frame).
- `selectedElementIds.length >= 1` → element scope (capture the
  *first* selected element). Multi-select doesn't get
  multi-export — the panel just shows the first one's name as
  the target.

The scope determines the size readout, the suggested filename
("home.png" vs "rect_a1b2.png"), and the DOM node
`exportCapture` operates on. No explicit "scope picker" in the
UI — the user's selection state IS the picker.

### Trigger surfaces

The section is always visible, so there's no "open the panel"
flow. The remaining triggers are all "do the export now" actions:

1. **Keyboard shortcut** — `ProjectShell.tsx`'s key handler
   gains a `Cmd+Shift+E` branch that calls the same export
   function the section's button calls, using the current
   saved settings (`lastFormat` / `lastPngScale`) and the
   current scope (selection-derived). Skips the panel UI
   entirely — just kicks off the save dialog.

2. **Element right-click** — `ElementRenderer.tsx`'s root gets
   an `onContextMenu` handler that selects the right-clicked
   element AND scrolls the Export section into view in the
   properties panel. The user lands with the right scope and
   the panel highlighted, ready to click Export. We use the
   existing `PageContextMenu` primitive with a single
   "Export…" item; the item's handler does
   `selectElement(id)` + `scrollExportSectionIntoView()`. Empty
   canvas right-click does nothing (out of scope).

3. **Toolbar button** — *removed from v1.* With the section
   always visible at the bottom of the WYSIWYG panel, a toolbar
   shortcut to "open it" is redundant. Cmd+Shift+E covers the
   "I want to export now" case. Listed under open questions
   in case the user disagrees.

### Export section — `ExportSection.tsx`

A new section component, dropped at the bottom of the panel
sections list — same `Section` / `Row` primitives every other
section uses (Background, Border, Shadow, …). Mounts from:

- `UiPanel.tsx` — when an element is selected, after
  `AnimationSection`.
- `PropertiesPanel.tsx`'s empty-state branch — when nothing is
  selected, replace (or accompany) the keyboard shortcuts
  table so the page-scoped Export section is reachable. Either
  scrolling shortcuts above and Export below, or showing
  Export at the bottom and a smaller shortcuts list above —
  see open questions.

ASCII layout:

```
┌─ Export ────────────────────── ┐
│  Format   [ PNG     ▾ ]        │   ← EnumSelect dropdown
│                                │
│  Scale    [1×] [2×] [3×]        │   ← SegmentedControl, PNG only
│                                │
│  Size     1440 × 900px          │   ← live readout
│                                │
│  [          Export          ]   │   ← primary button
└────────────────────────────────┘
```

Format is a dropdown (`EnumSelect`) per the user's spec:
`PNG / SVG / PDF`. PNG is the only format with a scale picker
(SegmentedControl 1× / 2× / 3×, default 2×). SVG and PDF show
no other options — SVG is resolution-independent, PDF is always
canvas-sized. The size readout shows:

- PNG: `(width × scale) × (height × scale)px`.
- SVG: `width × heightpx` (resolution-independent — vector).
- PDF: `width × heightpx` (canvas dimensions verbatim).

The Export button kicks off the format-specific flow (below).
The button label changes with scope to make the target obvious:
"Export page" / "Export rect_a1b2".

### Capture pipeline — `src/renderer/src/lib/exportCapture.ts`

A pure-ish module (no React, no store reads — takes the inputs as
arguments) that wraps `html-to-image` and the chrome-filtering
logic. Exports:

```ts
type CaptureInputs = {
  /** The DOM element to capture: the canvas frame for `kind: 'page'`,
   *  the element node (located by data-element-id) for `kind: 'element'`. */
  node: HTMLElement;
  /** Background color: 'transparent' for elements without a background,
   *  or the page background color. PNG / SVG only — PDF inherits the
   *  printed page's white background. */
  backgroundColor: string | null;
  /** Width and height in CSS px at 1×. The capture function multiplies
   *  by scale internally for PNG. */
  width: number;
  height: number;
};

export const capturePng = (
  inputs: CaptureInputs & { scale: 1 | 2 | 3 }
): Promise<string>; // data URL

export const captureSvg = (inputs: CaptureInputs): Promise<string>;
```

Both functions pass an `html-to-image` `filter` callback that
returns `false` for any node whose closest ancestor (including
itself) has `data-canvas-chrome="true"`. This is the single
opt-in marker we add to the chrome subtrees:

- `CanvasInteractionLayer` root → `data-canvas-chrome="true"`
- `SelectionOverlay` root → already inside the interaction layer,
  inherits the marker.
- `GridOverlay` root, `LinkIndicators` root → same.
- `ElementRenderer` and its descendants → no marker; included.

Adding the marker is a one-line edit per chrome component. Future
chrome additions inherit the convention by adding the attribute.

#### Zoom handling

The user might be zoomed in or out. The capture must produce the
image at intrinsic size, not at the on-screen size:

- **Page export** (`canvas-frame` node) — temporarily reset the
  frame's `transform: scale(...)` to `none` for the duration of
  the capture, then restore it. Since `html-to-image` reads the
  *applied* transform, we wrap the capture in a function that
  reads the current scale, sets it to 1, calls `htmlToImage.toPng`,
  and restores. Done synchronously in a single React render cycle.
- **Element export** — the element node inherits the frame's
  `transform: scale`, but `html-to-image` walks the subtree
  starting from the element's bounding box. Resetting the
  frame's scale before capture gives us intrinsic dimensions.

This trick is well-documented in the html-to-image issue tracker —
the alternative (`pixelRatio` option) gives the right resolution
but the wrong intrinsic size when the source is transformed.

#### Element bounding box

For element-scoped capture, `html-to-image.toPng(node, …)` already
captures only the element + its children. We supply
`width: el.offsetWidth, height: el.offsetHeight` so the output is
tightly cropped — no surrounding canvas whitespace. Per the story:
"cropped tightly to the element's bounding box with no surrounding
whitespace." ✓

### PDF pipeline

PDF can't go through `html-to-image` — we use Electron's native
`printToPDF()` for crisp vector text. The renderer assembles a
self-contained HTML document and hands it to main:

```ts
const buildPrintHtml = (inputs: {
  /** Cloned subtree (page or element) HTML — element + descendants. */
  contentHtml: string;
  /** Theme tokens as :root CSS custom properties. */
  themeCssText: string;
  /** Page-level @keyframes blocks (so animations don't break layout). */
  keyframesCssText: string;
  /** Width / height in CSS px for canvas-size mode. */
  bodyWidthPx?: number;
  bodyHeightPx?: number;
}): string;
```

The rough HTML shape:

```html
<!DOCTYPE html>
<html>
<head>
  <meta charset="utf-8" />
  <style>
    :root { /* theme tokens */ }
    @page { margin: 0; }
    body { margin: 0; background: white;
           /* canvas-mode: fixed body dims */ }
    /* keyframes */
  </style>
</head>
<body>
  <!-- contentHtml: cloned subtree, with computed inline styles -->
</body>
</html>
```

Cloning the live DOM (rather than re-rendering through React)
keeps the print artifact byte-equivalent to what's on screen. The
clone strips chrome subtrees by skipping `[data-canvas-chrome]`
nodes — same convention as PNG/SVG capture. The renderer hands
the resulting HTML string to the main handler.

Main process then:

1. Opens an offscreen `BrowserWindow` sized to `widthPx` /
   `heightPx`.
2. `loadURL('data:text/html;charset=utf-8,' + encodedHtml)`.
3. After `did-finish-load`, calls `webContents.printToPDF({
   pageSize: { width: widthPx * 26.4583, height: heightPx * 26.4583 },
   landscape: false, printBackground: true })`. `printToPDF`
   takes its `pageSize.width` / `pageSize.height` in microns;
   `1 CSS px ≈ 26.4583 microns` at 96 dpi.
4. Writes the buffer to disk.
5. Destroys the window.

The canvas's portrait/landscape aspect ratio is whatever the
user designed at — we don't reorient.

## File-naming convention

Per the story spec:

- Page: `[page-name].png` / `.svg` / `.pdf`. The page name is the
  active page id (e.g. `home`).
- Element: `[element-class].png` / `.svg` / `.pdf`. The element
  class name is what `classNameFor(el)` already produces (e.g.
  `rect_a1b2`, `hero-card_a1b2`).

Both are passed as the `filename` to `chooseSavePath`; the user
can rename in the dialog.

## Tests

Most of the export pipeline is hard to unit-test (it touches the
Electron main process, opens dialogs, writes files). The plan
focuses on the parts that *can* be tested:

### Pure helper tests (`test/exportCapture.test.ts`)

- `buildPrintHtml(...)` — given inputs, the output string contains
  the theme CSS, keyframes, and content in the right order; doc
  type is correct; `body` width/height set when canvas-mode.
- `pdfPageSizeToMicrons('A4')` etc. — conversion sanity for the
  `printToPDF` page-size argument.
- `suggestedFilename({ scope, activePage, elements })` — the
  page-name vs class-name selection logic.

### Integration test (`test/integration/export.integration.test.ts`)

- Mock `window.scamp.exportChooseSavePath` to return a temp
  directory. Trigger the renderer's PNG flow against a JSDOM
  canvas containing one rect with a known background color.
  Assert: the renderer calls `exportPng` with a non-empty
  data-URL, and the data-URL decodes to a PNG that's at least
  the requested size.

  This test exercises the chrome-filter logic without touching
  Electron.

### Manual / smoke checklist

(Listed in the implementation order so testing tracks
implementation.)

- PNG export at 1×, 2×, 3× — file opens in a viewer, no chrome
  visible, transparent background preserved when no bg set.
- SVG export — file opens, recognisable layout, the warning copy
  surfaces in the panel.
- PDF export at A4 portrait, A4 landscape, canvas-size — pages
  match the chosen size, content fits, fonts are crisp.
- Element export crops tightly — surrounding canvas whitespace
  not in the file.
- Cmd+Shift+E with selection → exports the selected element at
  the section's current settings; without selection → exports the
  page. No modal opens; the save dialog appears directly.
- Mid-zoom export uses the intrinsic size (zooming in then
  exporting still produces a 1440×900 image, not a zoomed image).
- Cancel save dialog → no file written, no console error. The
  Export section's controls keep their last values for retry.

E2E (Playwright) tests are explicitly out of scope until the UI
stabilises — same trade the rest of the codebase makes.

   until PNG/SVG are solid.

8. **Tests.** Unit tests for the pure helpers; integration
   test for the PNG renderer flow with mocked IPC.

9. **Polish.** Inline warning copy for SVG, the size-readout,
   default-extension handling in the save dialog filter,
   error toast when an export fails, scope-aware Export button
   label ("Export page" vs "Export rect_a1b2").

10. **Docs.** Update `agent.md` if relevant (probably not — this
    is a UX feature, not a CSS feature). Add a brief mention to
    the keyboard-shortcuts panel in `PropertiesPanel.tsx`.

## Risks and edge cases

- **`html-to-image` and CSS-Module / scoped class names.** The
  library walks the DOM and inlines computed styles into a
  `<foreignObject>`-based SVG. Scamp's canvas elements use inline
  styles already (`ElementRenderer.tsx` computes everything via
  `style={...}`), so there's nothing to scope-resolve. The page-
  level `<style>` tag with keyframes (`CanvasKeyframes`) WILL
  need to ride along in the captured DOM — `html-to-image`
  preserves `<style>` children of the captured root by default.
- **Animations mid-flight.** A capture taken while an animation
  is running grabs a frame at the current `animation-play-state`
  position. The story flags this as expected ("animated states
  are not captured" — meaning we capture the rest state, not
  multiple frames). For a v1, we accept whatever frame is
  current. A small follow-up could pause animations for the
  capture and resume after.
- **Large pages.** Capture at 3× scale of a 1440×900 canvas
  produces a 4320×2700 image. Memory pressure is real on lower-
  end machines. The renderer can guard against this by warning
  if `width × height × scale²` exceeds a threshold (e.g. 32 MP),
  but a v1 just lets the browser try. Document the limit in a
  follow-up.
- **Off-screen / clipped elements.** An element positioned
  outside the canvas frame's bounds (negative left, beyond
  width) is still part of the render tree. Page export captures
  whatever the frame's bounding box covers, which is the
  intended behaviour — clipping matches what the user sees.
  Element export captures the element node + descendants
  regardless of clipping, which also matches expectations.
- **Chrome filter false positives.** If a future chrome
  component forgets the `data-canvas-chrome` marker, it'll show
  up in the export. Mitigation: a `data-testid` on the chrome
  layer + a Playwright test that exports a clean canvas and
  asserts no chrome is visible. Out of scope for v1.
- **`var(--token)` resolution in PDF.** Theme CSS custom
  properties are defined on the canvas frame element
  (`Viewport.tsx:170` — `...themeCssVars`). The print HTML
  builder must inject those tokens at `:root` so descendant
  `var()` references resolve. The plan above already handles
  this in `buildPrintHtml`.
- **Background-image URL resolution.** The canvas resolves
  relative `url("./img.png")` references against
  `scamp-asset://` (`ElementRenderer.tsx:308`). The PDF print
  HTML must do the same — image references in the print HTML
  need fully-qualified `scamp-asset://` URLs. Reuse the
  existing rewrite logic.
- **Save-dialog cancellation.** The user can cancel at any step.
  All four IPC methods return well-typed `canceled` /
  `ok: false` results so the renderer can no-op cleanly. No
  partial-write states.
